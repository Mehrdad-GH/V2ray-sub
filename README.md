# Telegram → V2Ray Subscription (GitHub Actions, چند کاناله)

هر ۳ ساعت یه بار، یه GitHub Action از چند کانال عمومی تلگرام کانفیگ‌های `vmess/vless/trojan/ss/ssr` رو از متن پیام‌ها استخراج می‌کنه و یه فایل ساب (base64 استاندارد) می‌سازه.

بدون سرور، بدون دیتابیس. هر اجرا مستقیم از کانال‌ها می‌خونه و فایل رو کامل بازسازی می‌کنه (تجمیعی نیست).

## دو حالت تنظیم تعداد کانفیگ

**۱. تعداد جدا برای هر کانال**
```
CHANNELS = test1:10,test2:20
```
یعنی از `t.me/test1` آخرین ۱۰ کانفیگ، از `t.me/test2` آخرین ۲۰ تا. خروجی نهایی اتحاد این دوتاست (حدود ۳۰ کانفیگ، منهای تکراری‌ها).

**۲. یه سقف کلی برای مجموع همه**
```
CHANNELS = test1,test2,test3
TOTAL_LIMIT = 300
```
یعنی از هر سه کانال جمع‌آوری کن، بعد از رو *تاریخ پیام* مرتب کن و فقط ۳۰۰ تای جدیدترین (از کل مجموع) رو نگه‌دار.

می‌تونی این دو حالت رو ترکیب هم بکنی: بعضی کانال‌ها limit خاص خودشون رو دارن، بقیه ندارن، و در آخر `TOTAL_LIMIT` یه سقف نهایی رو کل نتیجه می‌ذاره.

## راه‌اندازی

**۱. ریپو بساز و کد رو پوش کن**

⚠️ ریپو باید **public** باشه، چون لینک ساب از `raw.githubusercontent.com` میاد که فقط برای ریپوهای public بدون احراز هویت در دسترسه.

```bash
git init
git add .
git commit -m "init"
gh repo create telegram-v2ray-sub --public --source=. --push
```

**۲. متغیرها رو تنظیم کن**

تو ریپو برو `Settings → Secrets and variables → Actions → Variables` و اضافه کن:

| اسم | مقدار | مثال | ضروری؟ |
|---|---|---|---|
| `CHANNELS` | لیست کانال‌ها (با یا بدون تعداد) | `test1:10,test2:20` یا `test1,test2` | بله |
| `TOTAL_LIMIT` | سقف کلی مجموع کانفیگ‌ها | `300` | نه |
| `MAX_PAGES_PER_CHANNEL` | حداکثر تعداد صفحه‌ای که هر کانال چک میشه (هر صفحه ~۲۰ پیام) | `15` (پیش‌فرض) | نه |
| `SUB_FILENAME` | اسم فایل خروجی | `sub.txt` (پیش‌فرض) | نه |

**۳. اولین اجرا**

برو تب `Actions → Update V2Ray Subscription → Run workflow` و دستی اجراش کن. بعد از اون هر ۳ ساعت خودکار اجرا میشه.

**۴. لینک ساب**
```
https://raw.githubusercontent.com/<username>/<repo>/main/sub/sub.txt
```
همینو تو v2rayN / v2rayNG به‌عنوان Subscription URL بذار.

## نکات

- اگه یه کانال `limit` نداشته باشه ولی `TOTAL_LIMIT` هم ست نشده باشه، اسکرپر تا `MAX_PAGES_PER_CHANNEL` صفحه به عقب می‌ره و **همه‌ی** کانفیگ‌های پیدا شده رو نگه می‌داره - می‌تونه لیست بزرگ بشه، بهتره یا `limit` بدی یا `TOTAL_LIMIT` بذاری.
- کانفیگ‌های تکراری بین کانال‌ها حذف میشن (بر اساس متن دقیق کانفیگ)، نسخه‌ی با تاریخ جدیدتر نگه داشته میشه.
- چون هر ران فایل ساب رو کامل از نو می‌سازه (نه تجمیعی)، اگه یه کانفیگ از تاریخچه‌ی اخیر کانال خارج بشه (پیام‌های قدیمی‌تر حذف/جایگزین بشن)، خودش از لیست بعدی خارج میشه.
- چون ریپو public میشه، برای کمی امنیت از طریق پوشیدگی، `SUB_FILENAME` رو یه رشته تصادفی بذار به‌جای `sub.txt`.
- اگه یکی از کانال‌ها private بشه، فقط همون کانال اسکیپ میشه (خطاش لاگ میشه)، بقیه ادامه پیدا می‌کنن.
- گیت‌هاب workflow های زمان‌بندی‌شده رو تو ریپوهای بی‌فعالیت طولانی (حدود ۶۰ روز بدون commit) خودکار غیرفعال می‌کنه؛ اگه دیدی دیگه آپدیت نمی‌خوره، از تب Actions دستی `Run workflow` بزن.
